# Deep Feature Audit: "Custom MCQ Exam Creator"

---

## 1. Feature Overview

The **Custom MCQ Exam Creator** is a user-facing feature that allows authenticated students to generate custom MCQ exams by selecting specific chapters from a class/subject hierarchy. Users configure exam parameters (title, question count, time limit, negative marking), take the exam in a timed session with question palette navigation, and view results with per-question review.

**Entry Points:**
- Bottom nav "Exam" → Exam Center (`/exams`) → "কাস্টম MCQ পরীক্ষা তৈরি করুন" promo card
- User Dashboard → Custom Exam History widget → "পরীক্ষা তৈরি করুন"
- Direct URL: `/create-exam`

**Language:** Bengali (all UI text is in Bengali)

| Field | Type | Notes |
|-------|------|-------|
| id | String (cuid) | PK |

| examId | String | FK → Exam.id (CASCADE delete) |
| score | Float | Calculated server-side |

| idempotencyKey | String? | Unique — prevents duplicate submissions |

### Data Lifecycle

1. **Create:** `POST /api/create-exam` → Creates `Exam` + `ExamQuestion` records
2. **Read:** `GET /api/exams/:id` → Reads `Exam` + `ExamQuestion` + `MCQ`
3. **Submit:** `POST /api/exams/results` → Creates `ExamResult` (score computed server-side)
4. **Review:** `GET /api/exams/results/detail` → Reads `ExamResult` + `Exam` + `ExamQuestion` + `MCQ`
5. **History:** `GET /api/exams/my-exams` → Reads `Exam` + `ExamResult` (aggregated stats)
6. **Delete:** CASCADE on User/Exam deletion

## 7. Business Logic

### Exam Generation

**File:** `src/app/api/create-exam/route.ts`

1. User selects chapters → server queries `MCQ` table filtered by `chapterId IN [...]`
2. If user has no subscription, `isPremium: false` filter is applied
3. MCQs are fetched with `take: 30` (hard cap), ordered by `createdAt desc`
4. Questions are **shuffled randomly** (`sort(() => Math.random() - 0.5)`) then sliced to requested count
5. An `Exam` record is created with `ExamQuestion` join records

### Question Selection

- **Randomly shuffled** from the pool of available MCQs in selected chapters
- Limited to **max 30 questions** (server-enforced)
- No deduplication beyond the `@@unique([examId, questionType, questionId])` constraint — but since we `slice(0, questionCount)` after shuffle, duplicates can't occur from the same pool query
- **Cannot produce duplicate questions** within a single exam (same MCQ ID can't appear twice in `ExamQuestion` due to unique constraint)

### Difficulty

- The `MCQ` model has a `difficulty` field (`easy`, `medium`, `hard`)
- **NOT used in custom exam creation** — questions are selected purely by chapter, not filtered by difficulty
- No difficulty selection UI in `CreateExamPage`

### Question Count

- User-specified, **hard-capped at 30** (both client and server enforced)
- Server: `Math.min(requestedCount, maxQuestions)` and `Math.min(mcqs.length, userCount)`
- Client: `<Input type="number" min={1} max={30}>` and `Math.min(questionCount, 30)` in request body

### Time Limit

- User-specified in minutes (5-180 range on client)
- Default: `Math.ceil(questionCount / 2)` minutes if not provided
- Stored as `duration` (Int, minutes) on Exam
- Client converts to seconds: `duration * 60`
- Timer counts down in `useExamStore`; auto-submits when reaching 0

### Pass Mark

- **NOT implemented** — no pass/fail threshold exists anywhere in the code
- `ExamResultPage` shows a grade (A+/A/A-/B/C/F) based on percentage ranges, but these are purely display — no pass/fail state is stored or enforced

### Negative Marking

- User-configurable toggle + value (0.25-5, step 0.25)
- Stored as `negativeMarks` on Exam
- **Applied server-side during scoring:** `calculatedScore = correct × marksPerMcq - wrong × negativeMarks`
- Score floored at 0: `Math.max(0, calculatedScore)`

### Scoring

**File:** `src/app/api/exams/results/route.ts`

- Score is **always calculated server-side** — client sends raw `answers` map, server fetches correct answers from DB
- Formula: `score = (correct × marksPerMcq) - (wrong × negativeMarks)` (floored at 0)
- `totalMarks = totalQuestions × marksPerMcq`
- Percentage is calculated client-side as `(correct / total) × 100`

### Ranking

- **NOT implemented** for custom exams
- No leaderboard or ranking system for custom exam results
- The `ExamLeaderboardTab` component exists but is part of the MCQ Exam Package system, not custom exams

### Exam History

- Custom exams are listed via `GET /api/exams/my-exams` filtered by `creatorId`
- Shows: title, question count, duration, attempts count, highest/average score, last attempt
- Supports search by title, pagination (20 per page)

### Resume

- **PARTIALLY IMPLEMENTED** — the exam store is persisted to localStorage (`zustand/persist` with key `edu-exam`)
- If the user refreshes the page mid-exam, the `useExamStore` state (answers, timer) persists
- However, the `ExamSessionPage` will re-fetch exam data from API and re-initialize if `isExamActive` is false
- Timer continues from persisted `timeRemaining`
- **Risk:** If the exam ID changes or the store is cleared, the exam session breaks

### Retry

- From `ExamResultPage`: "আবার চেষ্টা করুন" button → resets store → navigates back to `exam-session` with same `examId`
- From `CreatorExamHistoryPage`: "পুনরায় চেষ্টা" button → navigates to `exam-session` with same `examId`
- **However:** The `POST /api/exams/results` has a unique constraint `@@unique([userId, examId])` — a user can only submit ONCE per exam
- Retry navigates to the exam session, but submission will return the existing result (idempotent)
- This means **retry is effectively broken** — the user can re-take the exam but their second submission is silently ignored

### Analytics

- From `CreatorExamHistoryPage`: Shows highest score, average score, attempt count, last attempt date
- From `CreatorExamResultReviewPage`: Full question-by-question review with correct/wrong/skipped filtering
- **No advanced analytics** (no per-chapter breakdown, no time analysis, no weakness identification)

## 8. State Flow

### Zustand Exam Store (`src/store/exam.ts`)

**Persisted to localStorage** (key: `edu-exam`)

```
State:
  currentExamId: string | null
  answers: Record<string, string>     // { questionId: "A" }
  questionIds: string[]
  timeRemaining: number               // seconds
  isExamActive: boolean
  examDuration: number | null         // minutes
  answerCount: number

Actions:
  startExam(examId, duration)         // resets answers, sets timer
  setAnswer(questionId, answer)       // stores answer, increments count
  setQuestionIds(ids)                 // stores question ID list
  setTimeRemaining(timeOrUpdater)     // decrements each second
  endExam()                           // sets isExamActive = false
  resetExam()                         // clears everything
```

### Flow:

1. `ExamSessionPage` mounts → fetches exam from API → calls `startExam()`
2. Timer runs via `setInterval` → `setTimeRemaining(prev => prev - 1)`
3. User selects answer → `setAnswer(questionId, key)` → auto-advances to next question
4. User submits → `endExam()` → navigates to result page
5. On auto-submit (timeout) → `handleSubmitRef.current()` → same submit flow

### Client Components vs Server Components

- **All exam components are Client Components** (`'use client'`)
- Next.js page wrappers are Server Components (for SEO metadata) but render client components
- No Server Components in the exam-taking flow

## 9. Security Review

| Check | Status | Detail |
|-------|--------|--------|
| Authentication | ✅ | All API routes require `verifyAuth` (except exam detail which is partially public) |
| Authorization | ✅ | `results/detail` checks `result.userId === auth.user.id` |
| Role Protection | ✅ | Answers only included for admins in exam detail |
| Premium Content Protection | ✅ | Server filters premium MCQs based on subscription/purchase |
| User Isolation | ✅ | Each user's exam and results are isolated by `creatorId`/`userId` |
| Session Validation | ⚠️ | Exam session is purely client-side (Zustand persist) — no server-side session tracking |
| Ownership Validation | ✅ | `creatorId` set on exam creation, `userId` set on result |
| Rate Limiting | ✅ | `apiLimiter` on POST routes |
| CSRF | ✅ | CSRF token required on POST routes (`x-csrf-token` header) |
| Input Validation | ⚠️ | Minimal — `chapterIds` validated as array, but no validation on `duration` range, `questionCount` range at API level |
| SQL Injection | ✅ | Prisma ORM — parameterized queries |
| XSS | ✅ | React auto-escapes; `RichContentRenderer` used for HTML content |
| Replay Attack | ✅ | Idempotency key + unique constraint prevent duplicate submissions |
| Tampering | ✅ | Score calculated server-side from stored correct answers |
| Time Manipulation | ⚠️ | `timeTaken` is client-reported (`initialDurationSeconds - timeRemaining`) — could be manipulated, but only affects the `timeTaken` field, not the score |
| Answer Tampering | ✅ | Answers are submitted but score is computed server-side |
| Score Tampering | ✅ | Server computes score from DB-stored correct answers |
| Direct API Access | ⚠️ | User can call `POST /api/create-exam` multiple times to create many exams — no rate limit on exam creation beyond general API limiter |
| Hidden Routes | ✅ | All routes are standard Next.js API routes |
| Exam Manipulation | ⚠️ | User can re-fetch exam with `showAnswers=true` after admin check — but answers are only included for admins |

### Security Issues Found

1. **`timeTaken` is client-reported** — Minor: only affects time tracking, not score
2. **No per-exam session tracking** — No server-side state for "exam in progress" — relies entirely on client persistence
3. **Retry is silently broken** — User can navigate back to exam session but can't actually submit a second result

## 10. Performance Review

| Issue | Severity | Detail |
|-------|----------|--------|
| N+1 Query in `/api/exams/:id` | Medium | Premium MCQ access checked individually in `Promise.all` loop (line 125-135) |
| Full hierarchy load | Low | `/api/hierarchy/metadata` loads ALL classes, subjects, chapters — may be large |
| Exam creation is blocking | Low | MCQ query + shuffle + create happen synchronously in one request |
| No caching on exam data | Low | Exam questions fetched fresh every time (no stale-while-revalidate) |
| localStorage persistence | Low | Exam state persisted to localStorage — could grow large with many answers |

## 11. Current Issues

### CRITICAL

1. **Retry/Broken Re-take Flow**
   - **Cause:** `@@unique([userId, examId])` on `ExamResult` means only one submission per user per exam
   - **Impact:** "আবার চেষ্টা করুন" (retry) button navigates to exam session but the second submission returns the existing result silently — user thinks they re-took but their new score isn't saved
   - **Affected:** `ExamResultPage.tsx:451-458`, `CreatorExamHistoryPage.tsx:259`, `api/exams/results/route.ts:52-60`
   - **Suggested Fix:** Either allow multiple results (remove unique constraint, store all attempts) or create a new exam copy on retry
   - **Priority:** P0
   - **Complexity:** Medium

### HIGH

2. **No Server-Side Exam Session Tracking**
   - **Cause:** Exam state is entirely client-side (Zustand persist)
   - **Impact:** User can manipulate timer by clearing localStorage and re-loading; no way to detect if user left and came back; no way to enforce "no going back" instruction
   - **Affected:** `store/exam.ts`, `ExamSessionPage.tsx`
   - **Suggested Fix:** Add server-side exam session (start timestamp stored in DB, validated on submit)
   - **Priority:** P1
   - **Complexity:** Medium

3. **N+1 Premium Access Check**
   - **Cause:** `Promise.all` loop over each premium MCQ for individual `checkContentAccess` calls
   - **Impact:** For exams with many premium MCQs, this creates O(N) database queries
   - **Affected:** `src/app/api/exams/[id]/route.ts:124-136`
   - **Suggested Fix:** Batch premium access check into a single query
   - **Priority:** P1
   - **Complexity:** Medium

### MEDIUM

4. **No Difficulty Filtering**
   - **Cause:** Custom exam creation doesn't expose difficulty selection
   - **Impact:** Users cannot create exams of specific difficulty levels despite MCQ model having a `difficulty` field
   - **Affected:** `CreateExamPage.tsx`, `api/create-exam/route.ts`
   - **Suggested Fix:** Add difficulty filter to the creation form and pass it to the API
   - **Priority:** P2
   - **Complexity:** Low

5. **Duplicate Badge in CustomExamHistory**
   - **Cause:** Two identical `<Badge>` elements rendered (lines 99-106)
   - **Impact:** Visual redundancy
   - **Affected:** `src/components/user/dashboard/CustomExamHistory.tsx:99-106`
   - **Suggested Fix:** Remove the duplicate badge
   - **Priority:** P2
   - **Complexity:** Trivial

6. **`timeTaken` Can Be Negative**
   - **Cause:** `timeTaken = Math.max(0, initialDurationSeconds - timeRemaining)` — if `timeRemaining` somehow exceeds `initialDurationSeconds`, this works correctly, but the calculation relies on the client-computed `timeRemaining`
   - **Impact:** Minor — time tracking could be inaccurate
   - **Affected:** `ExamSessionPage.tsx:367-368`, `api/exams/results/route.ts:100`
   - **Priority:** P2
   - **Complexity:** Low

7. **No Exam Deletion**
   - **Cause:** No API endpoint to delete custom exams
   - **Impact:** Users cannot remove exams they created; no admin cleanup mechanism
   - **Affected:** Missing endpoint
   - **Suggested Fix:** Add DELETE endpoint with ownership check
   - **Priority:** P2
   - **Complexity:** Low

8. **MCQ Query Uses `orderBy: { createdAt: 'desc' }`**
   - **Cause:** Custom exam fetches the 30 most recently created MCQs, not randomly sampled
   - **Impact:** Questions are biased toward recently added MCQs rather than being truly random from the full pool
   - **Affected:** `src/app/api/create-exam/route.ts:72`
   - **Suggested Fix:** Use `Math.random()` on full result set or use Prisma's raw query for random sampling
   - **Priority:** P2
   - **Complexity:** Medium

### LOW

9. **No Passing Grade Enforcement**
   - **Cause:** No pass/fail logic in the system
   - **Impact:** All results displayed equally — no indication of pass/fail
   - **Affected:** Display only
   - **Priority:** P3
   - **Complexity:** Low

10. **No Analytics Beyond Basic Stats**
    - **Cause:** Only highest score, average score, and attempt count tracked
    - **Impact:** Users can't see per-chapter performance, time trends, or improvement tracking
    - **Priority:** P3
    - **Complexity:** Medium

11. **Exam Store Not Reset on Logout**
    - **Cause:** Zustand persist keeps exam state in localStorage even after logout
    - **Impact:** Edge case: if user logs out during exam and logs back in, old state persists
    - **Affected:** `store/exam.ts`
    - **Priority:** P3
    - **Complexity:** Low

12. **No Loading State for Subscription Check**
    - **Cause:** Subscription check shows a spinner badge but the "Start Exam" button remains enabled during check
    - **Impact:** User could potentially click before subscription status is resolved
    - **Affected:** `CreateExamPage.tsx:452` — button disabled only by `creating` and `selectedChapterIds.length === 0`
    - **Priority:** P3
